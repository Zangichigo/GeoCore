# GeoCore API Philosophy

This document describes the principles used when designing the public API of GeoCore.

The API is considered one of the project's most valuable assets.

A stable, intuitive and predictable API is preferred over a large or feature-rich one.

---

# Readability First

GeoCore APIs should be understandable without reading documentation.

Good examples:

```cpp
Math::distance(a, b);

Math::bearing(a, b);

Math::midpoint(a, b);

circle.contains(position);
```

The purpose of a function should be obvious from its name.

---

# Simple Names

Prefer short, explicit names.

Good:

```cpp
distance()

bearing()

midpoint()

contains()
```

Avoid unnecessary verbosity.

Bad:

```cpp
calculateDistanceBetweenPositions()

computePolygonContainment()
```

---

# Objects Represent Concepts

Classes represent geospatial concepts.

Examples:

- Position
- PositionSample
- Track
- Circle
- Rectangle
- Polygon

Objects primarily contain data and expose behavior that naturally belongs to them.

Example:

```cpp
circle.contains(position);
```

---

# Algorithms Belong to Modules

Algorithms are implemented as free functions inside dedicated namespaces.

Examples:

```cpp
Math::distance()

Math::bearing()

Math::midpoint()

Movement::speed()

Tracking::distance()
```

This separation keeps objects lightweight and responsibilities clear.

---

# Avoid God Objects

Objects should not accumulate unrelated functionality.

Good:

```cpp
Track
```

stores a trajectory.

Tracking

computes statistics.

Bad:

```cpp
track.distance();

track.averageSpeed();

track.exportGPX();

track.sendMQTT();
```

Responsibilities should remain separated.

---

# Const Correctness

Public APIs should expose const-correct interfaces whenever possible.

Example:

```cpp
double distance(
    const Position& a,
    const Position& b);
```

---

# Prefer Value Types

Small immutable objects should be passed by value when appropriate.

Larger objects should be passed by const reference.

---

# Explicit Types

GeoCore prefers explicit types over ambiguous ones.

Good:

```cpp
Position

Track

Circle
```

Avoid generic containers in public APIs whenever a meaningful type exists.

---

# Keep Dependencies Minimal

Each module should depend only on what it really needs.

Examples:

Math

does not depend on Tracking.

Geometry

does not depend on Movement.

Tracking

may depend on Math.

This keeps compilation fast and architecture clean.

---

# Public API Stability

Once released, public APIs should remain stable.

Breaking changes should remain exceptional.

When improvements are necessary, prefer extending APIs instead of replacing them.

---

# One Capability, One Purpose

Every public capability should solve one problem.

Examples:

Distance

calculates a distance.

Bearing

calculates a bearing.

Midpoint

returns a midpoint.

A capability should never attempt to solve multiple unrelated problems.

---

# Consistency Over Cleverness

GeoCore intentionally favors consistency over advanced language tricks.

Readable code is preferred over clever code.

Future contributors should understand the API immediately.

---

# Build for Humans

GeoCore is written for developers.

Every API should answer one question:

"Would a developer guess this API without reading the documentation?"

If the answer is yes, the API is probably well designed.
