# GeoCore Manifesto

## Why GeoCore Exists

GeoCore exists to make geospatial development simple.

Many excellent geospatial libraries already exist, but they are often designed for very broad domains, resulting in complex APIs and steep learning curves.

GeoCore follows a different philosophy.

Its goal is not to become the largest geospatial library.

Its goal is to become one of the most pleasant geospatial libraries to use for everyday development.

---

## Simplicity First

Every feature added to GeoCore must justify its existence.

A capability is accepted because it solves a common problem, not because another library already implements it.

GeoCore intentionally focuses on the 80% of use cases that developers encounter most often.

---

## Independence

GeoCore is completely independent.

It does not depend on:

- Home Assistant
- OpenFence
- GPS hardware
- Zigbee
- LoRa
- MQTT
- Cloud services

GeoCore only manipulates geospatial concepts.

Applications build on top of GeoCore—not the other way around.

---

## Objects and Algorithms

GeoCore separates data from behavior.

Objects represent information.

Examples:

- Position
- Track
- Circle
- Polygon
- Rectangle

Algorithms process these objects.

Examples:

- Distance
- Bearing
- Midpoint
- Speed

This separation keeps the API predictable and easy to extend.

---

## Hardware Agnostic

GeoCore never assumes where data comes from.

A Position may come from:

- GPS
- GNSS
- Simulation
- User input
- Database
- File
- Network

GeoCore does not care.

---

## Business Agnostic

GeoCore is not a geofencing library.

It is not a mapping library.

It is not a navigation library.

It is not a smart-home library.

It is a geospatial engine that any application can use.

---

## Formats Are Adapters

Formats such as GPX, GeoJSON or CSV are not part of the engine.

They are adapters that convert GeoCore objects into external representations.

This keeps the engine lightweight and focused.

---

## Stability Before Features

A simple, stable API is more valuable than a large API.

GeoCore prefers consistency over feature count.

Every new capability should improve the library without increasing unnecessary complexity.

---

## Our Goal

We want developers to discover GeoCore and immediately understand how to use it.

If a developer can become productive without reading hundreds of pages of documentation, then GeoCore has achieved its purpose.
