# GeoCore Design Principles

These principles guide every architectural and technical decision made within GeoCore.

When introducing a new capability, these principles always take precedence over convenience.

---

# 1. Simplicity First

GeoCore is intentionally simple.

A feature must solve a common problem before it deserves a place in the library.

If a capability makes GeoCore more difficult to understand than it makes it useful, it should not be added.

---

# 2. One Responsibility

Each class and each function should have one clear responsibility.

Examples:

- Position stores coordinates.
- Circle represents a circle.
- Track stores a trajectory.
- Distance computes a distance.

Responsibilities should never overlap.

---

# 3. Objects Hold Data

Objects represent geospatial concepts.

Examples:

- Position
- PositionSample
- Track
- Circle
- Rectangle
- Polygon

Objects should expose clear and intuitive APIs.

---

# 4. Algorithms Process Objects

Algorithms belong to dedicated modules.

Examples:

Math

- distance()
- bearing()
- midpoint()

Movement

- speed()
- Course()

Tracking

- distance()
- duration()

Objects should not become collections of unrelated methods.

---

# 5. Explicit APIs

GeoCore favors readability over cleverness.

Good:

distance(a, b)

Better than:

calculateDistanceBetweenPositions(a, b)

The API should be understandable without reading documentation.

---

# 6. Platform Independence

GeoCore must remain platform independent.

It should compile and behave consistently on every supported platform.

No platform-specific behavior belongs inside the engine.

---

# 7. Hardware Independence

GeoCore never depends on hardware.

Positions may originate from:

- GPS
- GNSS
- Simulation
- User input
- Database
- Network

The engine makes no assumptions.

---

# 8. Business Independence

GeoCore contains no business logic.

It should never become coupled to:

- OpenFence
- Home Assistant
- Asset tracking
- Navigation software
- Smart home systems

Applications depend on GeoCore.

GeoCore depends on nothing.

---

# 9. Formats Are Adapters

GPX, GeoJSON, CSV and similar technologies are adapters.

They translate GeoCore objects to external formats.

They are not part of the engine itself.

---

# 10. Test Everything

Every public capability should have:

- Unit tests
- Demonstration code
- Documentation

Reliability is a feature.

---

# 11. Backward Compatibility

Public APIs should evolve carefully.

Breaking changes should remain exceptional.

A stable API is more valuable than a rapidly changing one.

---

# 12. Build for the Next 10 Years

Every design decision should answer one question:

"Will this still make sense when GeoCore contains one hundred capabilities?"

If the answer is no, reconsider the design.
